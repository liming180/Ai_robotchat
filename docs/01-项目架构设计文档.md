## 🏗️ 一、前后端分离架构设计

### 1.1 架构理念

本项目采用**前后端完全分离**的现代化架构，遵循以下核心原则：

- **职责清晰**：前端专注于用户界面与交互体验，后端专注于业务逻辑与数据处理
- **独立部署**：前后端可独立开发、测试与部署，互不干扰
- **标准协议**：通过 RESTful API 与 WebSocket 进行通信
- **技术栈灵活**：前后端可采用各自最佳的技术栈

## 🔄 三、前后端通信规范

### 3.1 通信协议

- **RESTful API**：标准的 CRUD 操作
- **WebSocket**：实时聊天通信
- **数据格式**：JSON
- **字符编码**：UTF-8

### 3.2 基础 URL

| 环境 | 前端地址 | 后端 API 地址 | AI 服务地址 |
|------|----------|--------------|------------|
| 本地开发 | `http://localhost:3000` | `http://localhost:8080` | `http://localhost:5000` |
| 测试环境 | `https://test.example.com` | `https://api-test.example.com` |
| 生产环境 | `https://www.example.com` | `https://api.example.com` |

### 3.3 响应格式

#### 成功响应

```json
{
  "success": true,
  "data": { /* 响应数据 */ },
  "message": "操作成功",
  "timestamp": "2026-07-10T12:00:00Z"
}
```

#### 错误响应

```json
{
  "success": false,
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "参数验证失败",
    "details": [ /* 详细错误信息 */ ]
  },
  "timestamp": "2026-07-10T12:00:00Z"
}
```

### 3.4 状态码规范

| 状态码 | 说明 | 使用场景 |
|--------|------|----------|
| 200 | OK | 查询成功 |
| 201 | Created | 创建成功 |
| 400 | Bad Request | 参数错误 |
| 401 | Unauthorized | 未登录 / Token 过期 |
| 403 | Forbidden | 权限不足 |
| 404 | Not Found | 资源不存在 |
| 500 | Internal Server Error | 服务器内部错误 |

## 🔐 四、认证与授权方案

### 4.1 JWT 认证流程

```
1. 用户登录 -> 后端验证 -> 返回 access_token 和 refresh_token
2. 前端请求时在 Header 中携带: Authorization: Bearer {access_token}
3. 后端验证 JWT 并识别用户身份
4. Token 过期时使用 refresh_token 刷新
```

### 4.2 Token 策略

| Token 类型 | 有效期 | 用途 |
|-----------|--------|------|
| Access Token | 2 小时 | API 接口认证 |
| Refresh Token | 7 天 | 刷新 Access Token |

## 🌐 五、跨域解决方案

### 5.1 开发环境

使用 Spring Boot 的 CORS 配置，在开发时允许前端地址跨域：

```java
@Configuration
public class WebConfig implements WebMvcConfigurer {
    @Override
    public void addCorsMappings(CorsRegistry registry) {
        registry.addMapping("/**")
                .allowedOrigins("http://localhost:3000")
                .allowedMethods("*")
                .allowedHeaders("*")
                .allowCredentials(true);
    }
}
```

### 5.2 生产环境

使用 Nginx 反向代理，前端和后端在同一个域名下，避免跨域问题。

## 👥 六、协作开发流程

### 6.1 Git 分支策略

| 分支名 | 用途 | 说明 |
|--------|------|------|
| `main` | 主分支 | 稳定的生产代码，只接受 PR |
| `develop` | 开发分支 | 最新的开发代码 |
| `feature/*` | 功能分支 | 新功能开发，从 develop 分出 |
| `hotfix/*` | 热修复分支 | 紧急修复，从 main 分出 |

### 6.2 开发流程

1. **前端开发**：在 `frontend/` 目录下开发，Mock 数据或对接开发接口
2. **后端开发**：在 `backend/` 目录下开发，提供 API 文档
3. **联调测试**：功能完成后进行前后端联调
4. **代码审查**：提交 PR，通过后合并到 develop 分支

## 📝 七、配置文件说明

### 7.1 前端配置 (.env)

```env

VITE_API_BASE_URL=http://localhost:8080

# .env.production
VITE_API_BASE_URL=https://api.example.com
```

### 7.2 Java 后端配置 (application.yml)

```yaml
spring:
  application:
    name: ai-companion
  datasource:
    driver-class-name: com.mysql.cj.jdbc.Driver
    url: jdbc:mysql://localhost:3306/ai_companion?useUnicode=true&characterEncoding=utf-8&serverTimezone=Asia/Shanghai
    username: root
    password: your-password
  data:
    redis:
      host: localhost
      port: 6379
      database: 1

mybatis-plus:
  configuration:
    map-underscore-to-camel-case: true
  global-config:
    db-config:
      id-type: auto

jwt:
  secret: your-super-secret-key
  expiration: 7200000  # 2小时
  refresh-expiration: 604800000  # 7天

ai:
  service:
    url: http://localhost:5000
```

### 7.3 AI 服务配置 (.env)

```env
# 应用配置

LLM_MODEL=GLM-4.5-Air

# LangChain 配置
LANGCHAIN_TRACING_V2=false
LANGCHAIN_PROJECT=ai-companion
```

## 📊 九、数据库设计要点

- **MySQL 数据库**：
  - **用户表 (user)**：存储用户基本信息、认证信息
  - **伴侣表 (companion)**：存储预设和自定义 AI 伴侣
  - **对话表 (conversation)**：存储对话元数据
  - **消息表 (message)**：存储具体的聊天消息
  - **用户伴侣关联表 (user_companion)**：存储用户与伴侣的关系、收藏、亲密度等

- **Redis**：缓存会话、临时数据（过期时间 6 小时）
